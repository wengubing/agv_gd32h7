## 内存对齐笔记
●volatile关键字用于告诉编译器，一个变量的值可能以编译器无法预知的方式被改变。
●stm32 内存 用户全局对象从0x2000 0000 开始存放，每个地址对应8位数据寄存器，
●存放时，首地址按数据类型byte长度的整数倍开始！
●结构体时,是按照结构体中最大基本数据类型的整数倍开始存放的！
即保证访问结构体内部最长数据类型的地址是对齐的！
最长的基础数据类型？数组？
默认4字节对齐？
●当通过结构体指针访问内存时，其内部对象的存储地址就不是其类型长度的倍数了，但从首地址为0起的相对地址，依然是其类型长度的倍数，且访问不会引起异常！
●总结即：数据类型 决定其存储形式，其存储形式 决定了编译时分配的首地址！

stm32硬件错误中断排查经验：u64访问未对齐地址，导致硬件错误中断，mdk纯软件仿真可以模拟出来，且通过观察代码执行的颜色标记，
可以看到硬件错误中断发生的代码行，通过这个行号，可以快速定位问题代码！
其它方法定位发生硬件错误中断的代码行号：通过硬件错误中断的地址，找到对应的代码行号？

## 状态机编程经验：
1.尽量不要直接操作状态机状态，而是通过输入，在状态机内部处理状态转移
2.不要直接控制状态机的调用
3.内部的状态机，在外部状态退出时，要清除内部状态机的状态，以便下次进入时正常执行流程
4.除非状态机可以中断继续执行，不然所有状态机都统一处理为外部状态机退出时清除状态，以便下次进入时正常执行流程
（不对，状态机不一个被中断，应该自己退出！）
5.只能一条道走到底的状态机，输入页需要用状态机处理

## c++调用c函数
需要extern "C"{}，例：
extern "C"  {
  extern int SER_GetChar (void);
	extern void c_fuc_test(void);
}
c无法调用c++函数，c++可以调用c函数

c++要调c函数，需要extern "C"{}，例：
.h
#ifdef __cplusplus
extern "C" {
#endif

	void c_fuc(void);

#ifdef __cplusplus
}
#endif

.c
#include "c_Test.h"
void c_fuc(void)
{
	printf("Hello, c_fuc call ok! \n");
}
